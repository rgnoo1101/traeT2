# 03-原型开发任务清单与验收

## 1. 使用方式
本文件是直接执行清单。下一个智能体按“任务编号”顺序完成即可。

## 2. 里程碑
1. M1：完成登录 + 今日拍摄上传链路。
2. M2：完成匹配 + 互看 + 双向加好友。
3. M3：完成好友动态 + 私聊 + 相簿排序。
4. M4：完成联调、压测、问题收敛。

## 3. 任务清单（可执行）

### A. 项目初始化
A1. 配置 Firebase 项目与 Flutter 绑定
- 输入：Firebase 项目、平台配置文件。
- 输出：应用可成功连接 Firebase。
- DoD：应用启动后可完成 Auth 测试登录。

A2. 建立基础目录与模块骨架
- 输入：技术方案中的目录建议。
- 输出：features/auth/capture/match/friends/chat/album 模块目录。
- DoD：工程可编译，路由可进入空页面。

### B. 认证与用户
B1. 实现登录（原型可先匿名或手机号二选一）
- 输出：可拿到 uid 并写入 users。
- DoD：登录后主页可显示 uid/nickname。

B2. 用户资料基础字段落库
- 输出：users/{uid} 包含 timezone、createdAt。
- DoD：重复登录不重复创建脏数据。

### C. 每日通知与拍摄
C1. 今日状态机与 DailyFlowProvider
- 输出：idle/notified/capturing/uploaded/matching/matched/viewing/done/missed。
- DoD：状态切换在页面可见。

C2. 实现拍摄页倒计时 120 秒
- 输出：倒计时结束自动置 missed。
- DoD：超时后不可再提交。

C3. 上传照片至 Storage 并写 dailyShots
- 输出：shots 路径文件 + Firestore 元数据。
- DoD：上传成功后状态进入 uploaded。

### D. 匹配与互看
D1. 云函数 submitShot 与匹配池入队
- 输出：上传后进入可匹配队列。
- DoD：日志可观测入队成功。

D2. 云函数 createMatchIfPossible（随机+去重）
- 输出：matchSessions 文档生成。
- DoD：两个用户可被成功配对，最近 7 天不重复同人。

D3. 互看页 120 秒与会话读取
- 输出：双方照片可见、倒计时显示。
- DoD：倒计时结束后会话状态 expired/completed。

D4. 邀请逻辑 updateInviteDecision
- 输出：支持 yes/no；双 yes 触发 friendships。
- DoD：双 yes 后好友列表能看到对方。

### E. 好友与聊天
E1. 好友列表与好友今日照片
- 输出：friends 页面展示好友与其今日照片。
- DoD：新增好友后可见对方当天记录。

E2. 聊天会话与文本发送
- 输出：chats + messages 正常收发。
- DoD：双方可看到实时文本消息（或准实时刷新）。

### F. 相簿
F1. 个人相簿查询与排序
- 输出：按 localDate 分组，组内 timeOfDay 升序。
- DoD：时间顺序正确，跨天分组正确。

### G. 质量与验收
G1. 基础异常处理
- 输出：上传失败、匹配失败、网络中断有提示和重试。
- DoD：关键页面无静默失败。

G2. 并发压测（1000 上传）
- 输出：压测报告（成功率、P95、失败类型）。
- DoD：成功率 >= 99%，系统无崩溃。

G3. 回归清单执行
- 输出：主链路回归记录。
- DoD：通知→拍摄→匹配→互看→加好友→聊天→相簿全通过。

## 4. 最小测试用例
1. 正常链路：两用户同日拍摄后可匹配并互看。
2. 超时链路：通知后超过 2 分钟未上传，状态为 missed。
3. 邀请链路：单方 yes 不建好友，双方 yes 建好友。
4. 聊天链路：仅好友双方可读写聊天消息。
5. 相簿链路：当天照片按时间升序展示。
6. 去重链路：7 天内避免匹配到同一对象。

## 5. 开发优先级
1. P0：登录、拍摄上传、匹配、互看、双向加好友。
2. P1：好友动态、聊天、相簿。
3. P2：体验优化与埋点完善。

## 6. 风险与应对（原型级）
1. 风险：匹配函数竞争导致重复配对。
- 应对：使用事务或幂等锁字段。

2. 风险：上传高峰导致匹配延迟。
- 应对：异步队列化匹配，允许短暂等待页。

3. 风险：规则配置不当导致越权读写。
- 应对：先写最小白名单规则，再逐步放开。

## 7. 完成定义（项目级 DoD）
同时满足以下条件即视为原型完成：
1. 6 个功能需求全部可演示。
2. 核心数据结构已落地并可追踪。
3. 1000 并发上传压测达到目标口径。
4. 关键失败路径有提示与可恢复机制。

## 8. 交接执行指令（给下一个智能体）
按以下顺序执行：
1. 先完成 A-B-C（基础链路）。
2. 再完成 D（匹配互看与加好友）。
3. 再完成 E-F（社交与相簿）。
4. 最后完成 G（压测与验收）。

执行时严格以 P0 优先，任何新增需求先记录再评估，不阻塞主链路交付。
