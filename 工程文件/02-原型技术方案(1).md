# 02-原型技术方案

## 1. 技术选型
- 客户端：Flutter
- 状态管理：Provider
- 后端：Firebase 全家桶
  - Firebase Auth（登录）
  - Cloud Firestore（业务数据）
  - Firebase Storage（照片文件）
  - Cloud Functions（匹配与业务编排）
  - Firebase Cloud Messaging（每日通知）
  - Firebase Analytics + Crashlytics（基础监控）

## 2. 客户端架构（最小可实现）
建议分层：
1. presentation：页面与组件
2. state：Provider 状态对象
3. domain：实体与用例
4. data：Repository + Firebase 数据源

建议目录（位于 phototmatch/lib）：
- app/
- features/auth/
- features/capture/
- features/match/
- features/friends/
- features/chat/
- features/album/
- shared/

## 3. Provider 状态划分
1. AuthProvider
- 当前用户、登录态、登出。

2. DailyFlowProvider
- 今日状态：idle, notified, capturing, uploaded, matching, matched, viewing, done, missed。
- 倒计时控制（拍摄 2 分钟、互看 2 分钟）。

3. MatchProvider
- 当前匹配会话、对方用户、是否已发起邀请、对方邀请状态。

4. FriendProvider
- 好友列表、好友今日动态。

5. ChatProvider
- 会话列表、消息列表、发送状态。

6. AlbumProvider
- 个人相簿列表、日期分组与时间排序。

## 4. Firestore 数据模型（MVP）

### 4.1 users/{uid}
字段：
- uid: string
- nickname: string
- avatarUrl: string
- timezone: string
- createdAt: timestamp
- lastActiveAt: timestamp

### 4.2 dailyShots/{shotId}
字段：
- shotId: string
- uid: string
- localDate: string (YYYY-MM-DD)
- shotAt: timestamp
- timeOfDay: string (HH:mm:ss)
- storagePath: string
- status: string (uploaded/viewable/expired)

索引建议：
- (uid asc, shotAt desc)
- (localDate asc, shotAt asc)

### 4.3 matchSessions/{sessionId}
字段：
- sessionId: string
- userA: string
- userB: string
- localDate: string
- createdAt: timestamp
- viewExpireAt: timestamp
- state: string (active/expired/completed)
- userAInvite: bool
- userBInvite: bool
- becameFriends: bool

索引建议：
- (userA asc, createdAt desc)
- (userB asc, createdAt desc)
- (state asc, createdAt desc)

### 4.4 friendships/{friendshipId}
字段：
- uidLow: string
- uidHigh: string
- createdAt: timestamp
- sourceSessionId: string

唯一性建议：
- friendshipId = min(uid1,uid2)+'_'+max(uid1,uid2)

### 4.5 chats/{chatId}/messages/{messageId}
chats 字段：
- chatId: string（建议同 friendshipId）
- memberUids: string[]
- lastMessage: string
- lastMessageAt: timestamp

messages 字段：
- messageId: string
- senderUid: string
- text: string
- sentAt: timestamp

索引建议：
- messages 子集合按 sentAt 升序读取。

### 4.6 matchHistory/{uid}/records/{peerUid_date}
字段：
- peerUid: string
- matchedAt: timestamp
用于去重匹配（最近 N 天内不重复）。

## 5. Firebase Storage 结构
- shots/{uid}/{localDate}/{shotId}.jpg

约束：
- 上传后返回 storagePath 写入 dailyShots。
- 客户端仅读授权 URL，不直接暴露全路径列表。

## 6. Cloud Functions 设计（最小集合）
1. scheduleDailyNotification（定时任务）
- 输入：无
- 行为：按时区分桶，给当天应触发用户生成随机时点，并通过 FCM 发送通知。

2. submitShot（callable）
- 输入：uid, localDate, shotMeta/storagePath
- 行为：校验 2 分钟时限，写入 dailyShots，并将用户加入当日匹配池。

3. createMatchIfPossible（触发器/队列消费）
- 输入：匹配池新增事件
- 行为：随机+去重挑选对象，创建 matchSessions，写 matchHistory。

4. updateInviteDecision（callable）
- 输入：sessionId, uid, invite=true/false
- 行为：更新会话邀请状态；若双向 true，则创建 friendships 和 chats。

5. expireViewSession（定时清理）
- 输入：无
- 行为：将超时 session 标记为 expired。

6. sendChatMessage（callable，可选）
- 输入：chatId, text
- 行为：写入 messages，更新 chat 摘要。

## 7. 安全规则（原型级）
1. 所有读写需 authenticated。
2. 用户仅可写入自己的 dailyShots。
3. 用户仅可读取与自己相关的 matchSessions。
4. 聊天仅允许 memberUids 包含当前 uid 的会话访问。
5. friendships 只读给关系双方。

## 8. 1000 并发上传最小方案
1. 客户端：
- 压缩后上传（控制文件大小）。
- 上传失败自动重试 1-2 次。

2. 后端：
- 文件走 Firebase Storage，元数据走 Firestore，避免单点函数承压。
- 匹配逻辑异步化（通过触发器/队列），避免 submitShot 阻塞。

3. 数据层：
- 减少热点写入：匹配池按时区或哈希分片。
- 避免单文档高频更新。

4. 压测指标（原型验收口径）：
- 1000 并发上传时，上传成功率 >= 99%。
- P95 上传完成时间 <= 5 秒（在测试网络基线下）。
- 匹配结果产出延迟 P95 <= 10 秒。

## 9. 开发约束与默认值
1. 默认图片格式：jpg。
2. 默认去重窗口：7 天。
3. 默认互看时长：120 秒。
4. 默认拍摄时限：120 秒。
