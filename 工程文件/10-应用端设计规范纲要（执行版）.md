# 10-应用端设计规范纲要（执行版）

## 1. 文档信息
- 文档名称：PhotocMatch 应用端设计规范纲要（执行版）
- 文档版本：v1.0
- 更新时间：2026-04-10
- 关联文档：09 接口文档、06 拆分方案、07 本地验证方案

---

## 2. 目标与设计原则

### 2.1 目标
1. 将应用端固定为“展示 + 状态 + 调用”的瘦客户端。
2. 保障跨平台一致体验（Windows/iOS/Android/Web/macOS）。
3. 保证可在 `mock`、`real_local`、`real_cloud` 三模式切换。

### 2.2 原则
1. 单一职责：UI 不做业务终裁。
2. 契约优先：只按 09 文档字段交互。
3. 可回放：关键调用必须带 `traceId` 与 `requestId`（写接口）。
4. 可恢复：错误必须可分类、可提示、可重试。

---

## 3. 分层与目录规范

### 3.1 分层模型
1. Presentation：页面、组件、交互。
2. State：Provider/Notifier，仅维护状态与流程驱动。
3. Domain：实体、值对象、用例抽象。
4. Data：Repository 接口与实现、DTO、Mapper、Remote/Local DataSource。

### 3.2 推荐目录
```text
lib/
  app/
    bootstrap/
    routing/
    theme/
  features/
    auth/
    shot/
    match/
    friend/
    chat/
    album/
    notification/
  domain/
    entities/
    value_objects/
    repositories/
  data/
    repositories/
    datasources/
      remote/
      local/
    models/
    mappers/
  shared/
    error/
    network/
    time/
    logging/
    config/
```

### 3.3 依赖方向
1. Presentation -> State -> Domain -> Data
2. Domain 不依赖 Flutter UI 包。
3. State 不直接依赖 Firebase SDK，仅依赖 Repository 接口。

---

## 4. 关键禁区（必须遵守）
1. 禁止在 UI/Provider 直接调用 Firestore/Functions/Storage SDK。
2. 禁止在客户端裁决“能否匹配、能否建好友、是否超时有效”。
3. 禁止客户端直写核心集合：`matchQueue`、`matchSessions`、`friendships`。
4. 禁止用本地时钟作为业务时限依据。
5. 禁止跨层传递未封装异常（必须转换为统一错误模型）。

---

## 5. Repository 与数据源规范

### 5.1 Repository 约束
1. 每个业务域必须有接口：`AuthRepository`、`ShotRepository`、`MatchRepository`、`FriendRepository`、`ChatRepository`、`AlbumRepository`、`NotificationRepository`。
2. State 层只能依赖接口，不依赖具体实现。
3. 所有写方法必须要求 `requestId` 参数。

### 5.2 数据模式切换
1. `mock`：演示与页面开发。
2. `real_local`：对接 Emulator。
3. `real_cloud`：对接云端。

### 5.3 切换机制
1. 统一使用 `DATA_MODE` 配置（`--dart-define`）。
2. 模式切换不应导致页面层改动。
3. `real_local` 与 `real_cloud` 契约完全一致。

---

## 6. 网络调用与协议实现

### 6.1 请求规范
1. 统一客户端封装层发送请求。
2. 请求默认超时：连接 5 秒、读 15 秒、写 15 秒。
3. 写接口统一注入 `requestId`。

### 6.2 响应规范
1. 统一解析 `success/data/errorCode/message/traceId/serverTime`。
2. `traceId` 必须进入客户端日志上下文。
3. `serverTime` 每次响应后刷新 TimeManager。

### 6.3 重试规范
1. 可重试错误：`NETWORK_ERROR`、部分 `UNKNOWN_ERROR`。
2. 不可重试错误：`AUTH_REQUIRED`、`PERMISSION_DENIED`、`INVALID_REQUEST`。
3. 指数退避：1s -> 2s -> 4s（最多 3 次）。

---

## 7. 错误处理规范

### 7.1 统一错误模型
```text
AppError {
  code,
  message,
  traceId,
  retryable,
  recoverAction
}
```

### 7.2 显示策略
1. 业务可恢复错误：页面内提示 + 重试按钮。
2. 认证错误：重定向登录。
3. 权限错误：明确提示并阻断。

### 7.3 上报策略
1. 非预期错误上报日志系统（含 traceId、apiName、flowState）。
2. 用户态文案与开发日志文案分离。

---

## 8. 时间一致性规范

### 8.1 单一真值
1. 业务判定使用 `serverTime` 投影时间。
2. 本地时钟仅用于 UI 展示。

### 8.2 TimeManager 要求
1. 保存最近一次 `serverTime` 与本地接收时刻。
2. 提供 `now()` 供业务层读取。
3. 偏差超过阈值（5 秒）时记录告警日志。

### 8.3 业务规则
1. 120 秒拍摄时限以服务端校验为准。
2. 互看过期以 `viewExpireAt` + `serverTime` 判定。

---

## 9. 离线与缓存规范

### 9.1 本地缓存
1. 缓存用户资料、会话列表、好友列表、最近消息。
2. 缓存数据必须可失效与重拉。

### 9.2 离线队列
1. 仅允许消息发送类可离线排队（可选）。
2. 队列重放必须复用原 `requestId` 防重。
3. 队列失败上限后转人工重试。

### 9.3 图片处理
1. 上传前压缩（质量与尺寸可配置）。
2. 上传路径严格使用 `shots/{uid}/...`。
3. 上传失败要有重试与取消能力。

---

## 10. 状态管理规范

### 10.1 Provider 责任
1. 管理页面状态（loading/success/error/empty）。
2. 组织调用顺序与 UI 反馈。
3. 不承担业务终裁。

### 10.2 状态同步
1. 远端状态优先覆盖本地状态。
2. 乐观更新仅用于提升体验，必须可回滚。
3. 同步失败要恢复到服务端确认态。

---

## 11. 跨平台适配规范

### 11.1 能力适配层
1. 相机
2. 通知
3. 权限
4. 文件系统

### 11.2 适配要求
1. 平台差异限制在 adapter 层。
2. 业务流程层不写平台分支。
3. 保证 iOS/Android/Windows 行为一致。

---

## 12. 日志与可观测性规范

### 12.1 最小日志字段
1. `traceId`
2. `uid`
3. `platform`
4. `appVersion`
5. `apiName`
6. `flowState`
7. `latencyMs`
8. `result`
9. `errorCode`

### 12.2 关键埋点
1. 登录成功/失败
2. 拍摄提交成功/失败
3. 匹配会话拉取
4. 邀请决策
5. 消息发送

---

## 13. 安全与隐私规范
1. token 安全存储（平台安全存储组件）。
2. 禁止在日志中打印敏感隐私（完整 token、私密图片 URL）。
3. 权限申请文案必须说明用途。
4. 上传前做基础类型校验（jpg/jpeg）。

---

## 14. 测试与质量门禁

### 14.1 必测类型
1. 单元测试：Repository、TimeManager、ErrorMapper。
2. 集成测试：`real_local` 对接 Emulator。
3. Widget 测试：核心流程页面。
4. 回归测试：通知 -> 拍摄 -> 匹配 -> 互看 -> 好友 -> 聊天 -> 相簿。

### 14.2 门禁
1. `flutter analyze` 必须通过。
2. 核心流程测试必须通过。
3. 新增接口必须补充错误处理分支测试。

---

## 15. 构建与发布注意事项

### 15.1 构建前检查
1. `DATA_MODE` 配置是否正确。
2. API 版本是否与服务端一致。
3. Feature Flag 默认值是否符合目标环境。

### 15.2 Windows 本地联调
1. 先启动 Firebase Emulator。
2. 再运行 Flutter：
```powershell
flutter run -d windows --dart-define=DATA_MODE=real_local
```

### 15.3 多平台发布
1. iOS/Android 发布前必须完成权限与推送链路真机验证。
2. 禁止在发布包中保留 debug endpoint。

---

## 16. 变更管理
1. 接口字段遵循只增不改。
2. 重大变更需同步更新 09/11 文档。
3. 变更必须附带回归清单与回滚策略。
