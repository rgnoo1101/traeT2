## 3. 环境与调用入口

### 3.1 环境定义
1. `real_local`：Firebase Emulator（本地联调）
2. `real_cloud`：Firebase 云端（dev/test/staging/prod）

### 3.2 Callable 约定
1. 接口风格：Firebase Callable Functions（主）
2. 应用端统一通过 Repository 调用，不允许页面层直接调用 SDK。
3. 所有写接口必须携带 `requestId`。

### 3.3 本地端口（默认）
1. Functions：`127.0.0.1:5001`
2. Firestore：`127.0.0.1:8080`
3. Auth：`127.0.0.1:9099`
4. Storage：`127.0.0.1:9199`

---

## 4. 认证与鉴权

### 4.1 认证方式
1. Firebase Auth ID Token。
2. 服务端在每次调用前做 token 校验。

### 4.2 授权总原则
1. 默认拒绝，最小权限放行。
2. 核心集合（`matchQueue`、`matchSessions`、`friendships`）禁止客户端直写。
3. 关键业务裁决只允许服务端执行。

### 4.3 会话异常
1. token 失效返回 `AUTH_REQUIRED`。
2. 客户端收到后必须触发登录刷新流程，不做盲重试。

## 5. 通用协议规范

### 5.1 统一响应结构
```json
{
  "success": true,
  "data": {},
  "errorCode": "",
  "message": "",
  "traceId": "trc_xxx",
  "serverTime": "2026-04-10T08:00:00.000Z"
}
```

### 5.2 字段命名规范
1. JSON 使用 `camelCase`。
2. 主键统一 `*Id` 后缀，如 `shotId`、`sessionId`。
3. 时间统一 ISO8601 UTC：`YYYY-MM-DDTHH:mm:ss.SSSZ`。
4. 布尔建议 `is*`/`has*`。

### 5.3 时间一致性规范
1. 业务时限判定仅基于 `serverTime`。
2. 客户端本地时间仅用于展示。
3. 若本地时间与 `serverTime` 偏差超过 5 秒，客户端记录警告并触发校准。

### 5.4 幂等规范
1. 写接口必填 `requestId`。
2. 推荐格式：`req_<uid>_<uuidv4>`。
3. 幂等窗口：默认 24 小时（可配置）。
4. 同 `requestId` 重复调用，返回首次成功快照。

### 5.5 分页规范
1. 使用游标分页：`cursor + limit`。
2. `limit` 默认 20，最大 100。
3. 返回 `nextCursor`，无更多数据时为 `null`。

### 5.6 错误码与恢复动作
| errorCode | 含义 | 客户端动作 |
|---|---|---|
| AUTH_REQUIRED | 未登录或认证失效 | 跳转登录并刷新 token |
| TIME_WINDOW_EXPIRED | 120 秒拍摄时限过期 | 提示过期，不重试 |
| MATCH_NOT_FOUND | 匹配会话不存在 | 刷新会话列表，可重试 |
| PERMISSION_DENIED | 越权访问 | 终止操作并提示权限不足 |
| INVALID_REQUEST | 参数或格式不合法 | 修正参数后重试 |
| IDEMPOTENCY_CONFLICT | 幂等冲突 | 拉取首次结果快照 |
| UNKNOWN_ERROR | 未分类服务端错误 | 记录 traceId 并有限重试 |

## 7. 接口明细（按域）

## 7.1 Auth

### 7.1.1 auth.getCurrentUser
- 用途：获取当前登录用户资料与服务端时间基线。
- 请求：
```json
{
  "apiVersion": "v1"
}
```
- 响应 `data`：
```json
{
  "user": {
    "uid": "u_001",
    "nickname": "Alex",
    "avatarUrl": "https://...",
    "timezone": "Asia/Shanghai"
  }
}
```
- 常见错误：`AUTH_REQUIRED`
- 幂等：只读接口，无需 `requestId`。

## 7.2 Shot

### 7.2.1 shot.submitShot
- 用途：提交拍摄结果，进入匹配队列。
- 请求：
```json
{
  "apiVersion": "v1",
  "requestId": "req_u_001_3d7c2f6f-6a53-4ab0-a6ca-4a8a6281a001",
  "localDate": "2026-04-10",
  "shotAt": "2026-04-10T08:03:20.120Z",
  "storagePath": "shots/u_001/2026-04-10/img_001.jpg"
}
```
- 校验：
1. 调用用户必须与 `storagePath` 中 uid 一致。
2. `shotAt` 必须在服务端窗口内。
- 响应 `data`：
```json
{
  "shotId": "shot_001",
  "queueId": "mq_001",
  "status": "uploaded"
}
```
- 常见错误：`TIME_WINDOW_EXPIRED`、`INVALID_REQUEST`、`IDEMPOTENCY_CONFLICT`
- 事务副作用：递增 `counters/global.matchSequence` 并写入 `matchQueue`。

## 7.3 Match

### 7.3.1 match.getCurrentSession
- 用途：查询当前活跃匹配会话。
- 请求：
```json
{
  "apiVersion": "v1"
}
```
- 响应 `data`：
```json
{
  "session": {
    "sessionId": "ms_1001",
    "userA": "u_001",
    "userB": "u_002",
    "state": "active",
    "viewExpireAt": "2026-04-10T08:40:00.000Z",
    "userAInvite": null,
    "userBInvite": null
  }
}
```
- 常见错误：`MATCH_NOT_FOUND`

### 7.3.2 match.updateInviteDecision
- 用途：提交邀请决策（同意/拒绝）。
- 请求：
```json
{
  "apiVersion": "v1",
  "requestId": "req_u_001_2f7d91c2-18b8-48f5-8f03-f1da1000a777",
  "sessionId": "ms_1001",
  "decision": "accept"
}
```
- 规则：
1. 用户只能写自己对应的邀请位。
2. 双方都 accept 时，服务端事务创建 `friendships` 与 `chats`。
- 响应 `data`：
```json
{
  "sessionId": "ms_1001",
  "userDecision": "accept",
  "sessionState": "completed",
  "becameFriends": true,
  "friendshipId": "u_001_u_002"
}
```
- 常见错误：`PERMISSION_DENIED`、`MATCH_NOT_FOUND`、`IDEMPOTENCY_CONFLICT`

### 7.5.3 chat.sendMessage
- 用途：发送文本消息。
- 请求：
```json
{
  "apiVersion": "v1",
  "requestId": "req_u_001_6f1f3b96-2dc0-4fd1-997e-cc1b11df0088",
  "chatId": "u_001_u_002",
  "text": "Hello"
}
```
- 校验：
1. 调用用户必须属于 `memberUids`。
2. 文本长度上限 1000（可配置）。
- 响应 `data`：
```json
{
  "messageId": "msg_002",
  "chatId": "u_001_u_002",
  "sentAt": "2026-04-10T09:00:30.000Z"
}
```
- 常见错误：`PERMISSION_DENIED`、`INVALID_REQUEST`、`IDEMPOTENCY_CONFLICT`
- 事务副作用：写 `messages` 并更新 `chats.lastMessage*`。

## 8. 状态机约束

### 8.1 主流程状态
`idle -> notified -> capturing -> uploaded -> matching -> matched -> viewing -> done`

### 8.2 异常分支
1. `capturing -> missed`
2. `matching -> uploaded`
3. `viewing -> expired`

### 8.3 客户端限制
1. 客户端只能展示状态，不得自行推进关键状态。
2. 状态最终值以服务端响应为准。

## 9. 安全与权限矩阵（接口层）

| 域 | 客户端可读 | 客户端可写 | 备注 |
|---|---|---|---|
| users | 本人资料与必要公开信息 | 本人有限字段 | 受 rules 限制 |
| dailyShots | 本人 | 本人上传元数据 | 状态变更仅后端 |
| matchQueue | 否 | 否 | 后端独占 |
| matchSessions | 会话双方可读 | 仅本人邀请位 | 其他字段只读 |
| friendships | 双方可读 | 否 | 后端创建 |
| chats/messages | memberUids 可读 | 通过接口写消息 | 不可改成员 |
| notificationPlans | 本人简要只读 | 否 | 后端调度 |

## 10. 性能与限流建议

### 10.1 目标
1. 上传成功率 >= 99%
2. 上传 P95 <= 5 秒
3. 匹配 P95 <= 10 秒

### 10.2 限流建议
1. 单用户写接口 30 req/min
2. `chat.sendMessage` 60 req/min
3. 限流触发返回 `NETWORK_ERROR` 或扩展码（后续可引入 `TOO_MANY_REQUESTS`）

---

## 11. 本地与云端兼容约束
1. 同接口名、同请求结构、同响应结构。
2. 同错误码语义。
3. 本地仅替换基础设施，不改变业务字段。
4. `serverTime`、`requestId`、`traceId` 语义一致。

## 12. 构建与联调注意事项

### 12.1 应用端
1. 通过 `--dart-define=DATA_MODE=real_local|real_cloud` 切换。
2. 本地联调必须开启 Auth/Firestore/Storage/Functions Emulator。

### 12.2 服务端
1. Node.js 20 LTS，Java 17，Firebase CLI 最新稳定版。
2. 所有写接口先做幂等与权限校验，再进入事务。

### 12.3 测试顺序
1. Auth
2. Shot 提交
3. FIFO 匹配
4. 邀请建链
5. Chat
6. Album
7. Notification

## 13. 变更管理
1. 字段变更遵循“只增不改”。
2. 删除或重命名字段需提升 `apiVersion`。
3. 变更必须同步更新 10/11 规范文档与回归用例。
